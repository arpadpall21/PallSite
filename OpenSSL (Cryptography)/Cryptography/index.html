Description and Demonstration :

Cypher / Decypher

- It's basically a way encryting and decryption some data

Block cypher VS Stream cypher

- Block Cypher = encrypting one block at the time (ex: takes one 8 byte block at the time to encrypt, if the block is not a full 8 byte it will be padded)

- Stream Cypher = encrypting one bit or byte at the time (encrypting the data as it streams)

Encryption Modes / AEAD

- Block Ciphers can have encryption modes, the mode basically describes which 'mode' these blocks are operating with each other

- Encryption Modes can provide a extra security (CBC), streaming, authenticity (GCM) and more

- the AEAD (Athenticated Encryption Associated Data) is an encryption mode (algorithm) (like: GCM, CCM, OBC) where we can pass additional data to the encryption which also provides authenticity to the encryption

Symmetric VS Asymmetric encryption

- Symmetric encryption uses one "symmetric" key for encryption and decryption (fast encryption)

- Asymmetric encryption uses 2 keys, public key for encryption and private key for decryption (slow encryptions) (the public key can only encrypt the data but cannot decrypt it)

Sign / Verify

- The sender "signs" (also encodes) a file (data) with the private key (which is never transfered)

- The receiver "verifies" (also decodes) the signed data with the public key pair

- This technique assures that the receiver gets the original data from the intended sender (Data integrity and authenticity)

Hashing

- Hashing is a one-way encryption technique, we use a hashign function (hasing algorithm) and the data in order to create a 'hash'

- the result hash has a lot of names => "hash", "digest", "checksum"

- SHA (Secure Hashing Algorithm) families are MD-5, SHA-0|1|2|3 (ex: SHA-256 is a SHA-2 family hashing algorithm)

- Hashing Characteristics :

- the generated has cannot be decrypted to get the original data back

- the generated hash has fixed length no matter the input data lenth (length depends on the hasing algoritm)

- small change in the original data changes drastically the hash

Salting

- Salting is adding some data (pading the data) to the original data then hashing it, this way the generated hash is even harder to guess

MAC / HMAC (key-Hashed / Message Authentication Code)

- MAC (Message Authentication Code) is basically a hash (digest, checksum) added to the message which assures the message Integrity and Authenticity (message has not been tempered with and comes from the intendes sender)

- HMAC (key-Hashed Message Authentication Code) - is a technique (algorithm) which uses a key + hash algorithm + the original data in order to generate a specific type of MAC

- HMAC is also refered as a hash that was generated with a HMAC technique

- there are also other tehcniques to generate specific types of MACs (ex: OMAC etc...)

Key Exchange Protocols DH / ECDH / RSA

- DH (Diffie Hellman) is a very popular key exchange protocol

- ECDH (Elliptic Curve Diffie Hellman) is the same protocol as the DH but internaly uses an elliptic curve (basically it'ss more secure and CPU efficient than DH)

- RSA (Rivest Samir Adleman) provides authenticity to the Diffie Hellman key exchange (sign/verify the exchanged public keys)

File Extensions

- *.pem (Privaci Enhanced Mail)

- ASCII encoded certificate, key (any), etc...

- in the file the stored data is between a header and footer (which describes it ex: -----BEGIN RSA PRIVATE KEY-----)

- the actual certificate, key, etc... between the header and footer is base64 encoded

- *.der (Distinguished Encoding Rules)

- binary encoded certificate, key (any), etc...

- *.cer or *.crt (CERtificate)

- DER encoded certificate (on Windows may accept PEM encoded data)

- *.pfx or *.p12

- binary encoded file which contains the server's certificate (and any intermediate certificates) and private key
