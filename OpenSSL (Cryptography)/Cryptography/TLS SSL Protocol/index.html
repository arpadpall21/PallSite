Description and Demonstration :

TLS / SSL

- the TLS/SSL protocol estabilishes an encrypted connection between two computers (server - client)

- TLS (Transport Layer Security) and SSL (Secure Sockets Layer) are the same (version history: SSLv2 -> SSLv3 -> TLS1 -> TLS1.1 -> TLS1.2 -> TLS1.3)

SSL Certificate

- an SSL Certificate is a digital certificate taht authenticates a web server, it was signed by a ICA (Intermediate Certificate Authority)

- a Certificate Authority (CA) checks (in different levels) the identity of the requesting organisation before signing its CSR (Certificate Signing Request)

- an SSL Certificate contains : issuing authority / AIA (Authority Infor Access (ICA informations)) / version (like: V3) / serial number / signature algorithm (like: sha256) / signature hash algorithm (the algorithm that hashes the certificate which is actually signed) / subject information (CA (Common Name), location, email, etc...) / valid dates (from - to) (max up to 39 months) / public key / etc..

Perfect Forward Secrecy

- the Perefect Forward Secrecy is an additional TLS feature which generates a new key pair for each TLS handshake (communication session) (ephemeral method)

- currently [2020-06-19] there are 2 methods are used to achive Perfect Forward Secrecy (DHE = Diffie Hellman) (ECDHE = Elliptic Curve Diffie Hellman) (the trailing E is appended for traditional abbreviations)

- Perferct Forward Secrecy is optional up to TLSv1.2, but all TLSv1.3 use the ECDHE Perfect Forward Secrecy method

Pre Shared Keys (PSK)

- the Pre Shared Keys encryption method uses Symmetric Encryption with pre-shared keys, it's supported by TLS but not recommended

Session Renegotiation

- TLSv1.2 (and below) allows the client or the server to renegotiate certain aspects of the TLS session, TLSv1.3 does not allow client renegotiation

Session Resumption

- the TLS handshake is relatively slow, so there are 2 ways to 'resume' a session

Session Identifiers (NOT supported by TLSv1.3)

- the Server generates a unique ID for each connection and sends it to the client, both the Client and Server save the session state

- when the Client wants to reconnect sends the ID of the saved session state to the Server

- if the Server has the same Session Id they can resume, otherwise the Server will create a new connection

Session Tickets

- the Server encrypts the entire session state and sends it to the client as a 'ticket'

- when the Client reconnects sends the state to the Server (no need for Serve-Side caching)
